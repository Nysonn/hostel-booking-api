# Hostel Booking API

A RESTful backend API for managing hostel accommodations around Ugandan universities. Universities register landlords, landlords list hostels and rooms, and students browse, book, and pay for accommodation вЂ” all coordinated through a single admin account.

---

## User Roles

| Role | Responsibilities |
|------|-----------------|
| **Admin** | Seeds the system. Manages all users (create universities, suspend/unsuspend/delete any account). |
| **University** | Registers landlords affiliated with their institution. Views their landlords, students, and hostels. |
| **Landlord** | Creates and manages hostels and rooms. Receives booking and payment notifications. |
| **Student** | Self-registers. Browses hostels, books a room, makes payments (partial or full), terminates bookings. |

---

## Prerequisites

- Docker and Docker Compose
- GNU Make
- Node.js 22+ (for local development without Docker)
- A Clerk account (for authentication)
- A Cloudinary account (for file uploads)
- A Resend account (for transactional email)
- A Firebase project with a service account (for push notifications)
- A PostgreSQL database (can be managed externally or via a cloud provider)

## Getting Started

### Running with Docker

The project ships with a multi-stage Dockerfile and a Docker Compose configuration.

**Start the development server (hot reload via nodemon/tsx):**

```bash
make dev
```

This builds the image targeting the `dev` stage and starts the `api` service. Source files are bind-mounted so changes are reflected immediately without a rebuild.

**Stop the server:**

```bash
docker-compose down
```

### Local Development (without Docker)

```bash
npm install
cp .env.example .env   # fill in values
npm run dev
```

## Database

### Running Migrations

Migrations are generated by Drizzle Kit and stored in `src/db/migrations/`. To apply them inside the running Docker container:

```bash
make migrate-up
```

To generate a new migration after editing any schema file:

```bash
npx drizzle-kit generate
```

The system requires one admin user seeded into both Clerk and the local database. The seed script is idempotent вЂ” running it multiple times is safe.

### Booking Routes

All booking routes require `role: student` and an active (non-suspended) account.

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| POST | `/api/student/bookings` | Student | Create a booking for an available room |
| GET | `/api/student/bookings` | Student | List all bookings for this student |
| GET | `/api/student/bookings/:bookingId` | Student | Get a single booking with room and hostel details |
| POST | `/api/student/bookings/:bookingId/terminate` | Student | Terminate an active booking |

**POST /api/student/bookings body:**
```json
{ "room_id": "uuid-of-room" }
```

Business rules enforced:
- The room must exist and have available slots.
- A student may not hold more than one active booking at a time.
- Terminating a booking decrements occupied slots and restores room availability if it was full.

### Payment Routes

All payment routes require `role: student` and an active (non-suspended) account.

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| POST | `/api/student/bookings/:bookingId/payments` | Student | Process a payment for a booking |
| GET | `/api/student/bookings/:bookingId/payments` | Student | List all payments for a specific booking |

**POST /api/student/bookings/:bookingId/payments body:**
```json
{
  "payment_method": "mtn_mobile_money",
  "payment_type": "partial"
}
```

Payment rules:
- `partial` pays 50% of the room price; only one partial payment is allowed per booking.
- `full` pays 100% of the room price.
- Once a full payment exists, no further payments are accepted.

## Architectural Decisions

### Authentication вЂ” Clerk

Clerk handles JWT issuance and session management. `clerkMiddleware()` is applied globally and processes every incoming request. The custom `requireAuth` middleware calls `getAuth(req)` to extract the Clerk user ID, then fetches the corresponding record from the local `users` table to get the role and suspension status. The local database is the authoritative source for `role` and `is_suspended`; Clerk's `publicMetadata` is used only at user-creation time.

Suspension is implemented using Clerk's `banUser`/`unbanUser` API in addition to the local flag, ensuring suspended users cannot obtain new sessions.

### Database вЂ” Drizzle ORM + PostgreSQL

The schema is defined code-first using `pgTable` and `pgEnum`. All PostgreSQL enum types are centralized in `src/db/schema/enums.ts` to prevent duplicate-type errors when multiple tables reference the same enum (e.g., `gender` is shared by `landlords` and `students`).

Drizzle represents `numeric` columns as `string` in TypeScript to avoid floating-point precision loss. When performing arithmetic on prices (e.g., 50% partial payment), the value is parsed to `float`, rounded to two decimal places, and converted back to `string` before insertion.

The `declaration` TypeScript compiler option is disabled because Drizzle's internal column builder types cannot be named in `.d.ts` files without referencing private exports.

#### Migration script (`src/db/migrate.ts`)

The custom migration script uses `@neondatabase/serverless` v1.1.0, which dropped the `sql(string, params)` conventional function call in favour of `sql.query(string, params)`. All migration queries use `sql.query()`.

Node.js's built-in `fetch` (undici) can time out connecting to Neon even when the host is reachable on port 443, because undici's DNS resolver may select an IP address that does not respond and does not fall back to the next one (unlike `curl`). To work around this, the script overrides `neonConfig.fetchFunction` with a custom implementation that:

1. Resolves all IPv4 addresses for the Neon host using `dns.resolve4()`.
2. Tries each address in sequence via `https.request`, setting the TLS `servername` to the original hostname for correct SNI.
3. Returns the first successful response, or throws if all addresses fail.

This keeps the runtime dependency on `@neondatabase/serverless` for all app code while making the migration script resilient to DNS/network environments that block port 5432 (the standard PostgreSQL TCP port) and where undici's fetch path fails silently.

### File Uploads вЂ” Cloudinary

Multer is configured with `memoryStorage()` so uploaded files never touch disk. The in-memory buffer is streamed directly to Cloudinary via `upload_stream`. This keeps the container stateless and avoids the need for a persistent volume for user-generated content.

### Email вЂ” Resend

All transactional emails are sent asynchronously after the primary database transaction completes. Email delivery failures do not affect the API response or trigger a rollback. Functions in `src/utils/email.ts` use fire-and-forget `.catch()` logging at the call site.

### Push Notifications вЂ” Firebase Admin SDK

In-app notifications are always written to the `notifications` table. Firebase Cloud Messaging push notifications are sent only when an FCM token is provided to `createNotification()`. Since token storage is out of scope for this version, the push path is always skipped but the code path is present and ready to be activated once a token storage strategy is added to the schema.

### Transaction Strategy

Operations that modify multiple tables (booking creation, booking termination) are wrapped in a single Drizzle transaction. Crucially, all re-validation (room availability, duplicate booking check, booking ownership and status) is repeated inside the transaction to prevent race conditions under concurrent requests. External service calls (Clerk, Cloudinary) are always performed outside database transactions to avoid holding a connection open during network I/O.

If a Clerk user is created successfully but the subsequent database transaction fails, the service layer attempts to delete the orphaned Clerk user before re-throwing the error.

### Side Effects Strategy

After a successful transaction (booking creation/termination, payment processing), emails and in-app notifications are dispatched using an unawaited `.then()` chain so the HTTP response is returned immediately. Inside the side-effect handler, `Promise.allSettled` is used so that a failing email does not prevent a notification from being created, and vice versa.

### Cascade Deletes

The database schema uses no `ON DELETE CASCADE` foreign key constraints. All cascading deletes are implemented explicitly in `src/modules/admin/admin.repository.ts` using a Drizzle transaction. The deletion order is: payments в†’ bookings в†’ rooms в†’ hostels в†’ landlord в†’ notifications в†’ user. For university deletions, all affiliated landlords and students are cleaned up recursively in the same transaction.

### Error Handling

All service-layer errors are plain `Error` instances with an optional `status` property (e.g., `Object.assign(new Error("Not found"), { status: 404 })`). The global error handler in `src/middlewares/error.ts` reads this property to set the HTTP status code, and returns a consistent `{ success: false, message }` envelope. `ZodError` and `MulterError` are handled with dedicated branches (422 and 400 respectively).
